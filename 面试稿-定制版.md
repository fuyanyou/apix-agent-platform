## 一、1 分钟项目介绍（开场必问）

> 「我做了一个叫 APIX 的企业级多智能体协作平台，是一个从零搭建的 LLM 应用系统。后端用 Python + LangGraph + FastAPI 实现了 5 个微服务，前端是 Electron 桌面客户端。
>
> 核心能力是：用户可以跟 AI 对话，AI 能调用文件读写、网页搜索、知识库检索等工具，还能自动把复杂任务拆解成子任务分派给多个子 Agent 并行执行，执行完汇总结果返回用户。整套系统支持多模型切换、对话记忆持久化、还能做 RAG 文档问答。
>
> 我一个人负责了全部架构设计和代码实现，开发周期大概 [X 个月]。代码量大概 [数千] 行 Python + 前端。」

**控制 60 秒内。** 录音自检：是否说清了「是什么、怎么做的、你负责什么」。

## 二、核心模块 STAR 展开（面试重点，每模块 2~3 分钟）

### 模块 1：Agent 运行时与多 Agent 协作

- **S（情境）**：市面上很多 LLM 应用只能一问一答，不能做复杂多步骤任务。我想让 Agent 像团队 Leader 一样，把任务拆解后分给子 Agent 执行。

- **T（任务）**：我负责设计整个 Agent 运行时，支持 Leader 派发任务、Worker 后台执行、Leader 异步接收结果的协作模式。

- **A（行动）**：
  - **图编译层**：用 LangGraph 的 StateGraph 搭节点图，`agent_node` → `tool_node` → `agent_node` 循环直到 `END`，每个编译好的图用 `hash_key` 做缓存（key = agent_name + config 序列化），避免高频 LLM 调用下重复编译。
  - **角色体系**：定义了 5 种角色 — 普通 agent、main_agent（可分配 1 个子 Agent）、sub_agent（工具受限）、team_leader（分配多子 Agent）、team_worker（共享生成缓存执行）。
  - **调度引擎**：Leader 通过 `team_task_manager.submit_task()` 把子任务放入 `asyncio.Queue`，后台 `sub-agent-worker` 协程阻塞消费队列，`asyncio.create_task` 启动子 Agent 协程独立执行。停止通道走独立 `asyncio.Queue`，发 `task.cancel()`。Leader 通过 WebSocket 心跳异步拿到结果，更新 TODO 列表并通知用户。

- **R（结果）**：这套架构支持了 Leader 同时派发多个子 Agent 并行工作，在本地测试中可以跑通多步骤文件分析、代码生成、知识检索等需要工具组合和任务拆解的复杂场景。

**追问预判**：
- Q: 子 Agent 失败了怎么办？→ A: `task_state_store` 标记 `failed`，心跳消息带异常信息反馈 Leader，Leader 可选择重试或告知用户。
- Q: 为什么用 asyncio.Queue 而不是消息队列？→ A: 架构演进考虑，当前阶段单进程 asyncio.Queue 足够，后续横向扩展时可替换为 Redis Stream 或 Kafka。

---

### 模块 2：LLM 适配与容错

- **S**：用户可能用不同的大模型（OpenAI、DeepSeek、Ollama 等），且国内网络不稳，需要应对超时、限流、token 超限等问题。

- **T**：我负责做统一 LLM 适配层，屏蔽不同厂商的 API 差异，并做异常分类和自动重试。

- **A**：
  - **统一接口**：`LlmNodeAdapter.astream()` 内部根据 `provider` 做差异化参数映射（如 DeepSeek 的 `reasoning` 参数处理方式跟 OpenAI 不同），上层调用者无需感知厂商差异。
  - **视觉能力检测**：内置一个极小 test image 的 base64 探针，首次调用时自动检测模型是否支持多模态，结果缓存到本地 JSON 文件，后续直接读取避免重复 API 开销。
  - **异常分类**：捕获 API 异常后通过关键词匹配分为 `token_exceed`（上下文过长 → 触发上下文压缩）、`rate_limit`（限流 → 退避重试）和 `others`（通用错误提示）。

- **R**：支持了 OpenAI / DeepSeek / MoonShot / Ollama / 自定义 共 4+N 种厂商接入。异常分类机制让系统在恶劣网络下能自愈，不会因为一次超时就整个对话中断。

**追问预判**：
- Q: 怎么处理流式输出中断？→ A: 流式中断时保留已输出的内容，通过 WebSocket `msg_stream_abort` 事件通知前端标记中断状态，用户可以看到已生成的部分内容。

---

### 模块 3：RAG 文档问答（FILE 服务）

- **S**：用户需要上传 PDF、Word 等文档让 AI 基于文档内容回答问题，不能只靠模型记忆。

- **T**：我负责 FILE 微服务全链路——文档解析 → 向量化 → 存储 → 相似检索。

- **A**：
  - **文档转换**：LibreOffice 把 Office → PDF，PyMuPDF 逐页转 JPEG，打包 ZIP 供前端预览，控制并发 Worker 数为 2 避免资源争抢。
  - **向量化**：Embedding 工厂模式（`get_embed_model`），按 provider + model 动态创建 embedding function，支持切换不同向量模型。
  - **Milvus 存储**：用 `partition_key_field="user_id"` 做**多租户隔离**，每个用户只能检索自己的文档。索引类型当前用 FLAT（暴力搜索保证精度），预留 HNSW 切换接口。
  - **检索接口**：支持指定 provider/model 参数，不同模型用不同向量维度索引，检索结果返回 Document 对象含 metadata（来源文件、页码）。

- **R**：跑通全链路——用户上传文档 → 后台自动向量化 → Agent 通过 `search_knowledge_base` 工具调用检索接口 → LLM 基于检索结果生成回答。

**追问预判**：
- Q: 为什么选 FLAT 而不是 HNSW？→ A: 当前数据量不大，FLAT 保证 100% 召回率且无索引构建开销。架构已预留切换空间，数据量上来后改成 HNSW。
- Q: 文档切分策略？→ A: 用 `text_spliter.py` 做固定长度切分，后续可迭代为语义切分。

## 三、行为面试题（各 1.5 分钟）

### 题 1：「项目中遇到的最大技术难点是什么？」

- **S+T**：做 SubAgent 异步调度时，Leader 通过工具调用 `assign_sub_assistant` 给子 Agent 派任务，但 Leader 自己的 LLM 生成流程不能阻塞等结果——需要做到「派完继续聊，结果异步回来」。

- **A**：
  - 三层解耦：① Leader 的 LangGraph 图中 `assign_sub_assistant` 工具只负责 `submit_task()` 把任务放入 `asyncio.Queue` 然后立即返回占位符；② 后台 `sub-agent-worker` 协程阻塞消费，启动新协程跑子 Agent 的 LangGraph 流，完成后更新 `task_state_store`；③ 全部子任务完成后，`event_pipe` 发 `on_generation_team_task_completed`，`feedback_to_agent` 用心跳消息唤醒 Leader，Leader 调用 `query_sub_assistant` 拿到结果更新 TODO。
  - 踩坑：最棘手的是 asyncio 协程取消的边界条件。用户在子 Agent 执行中点「停止」，需要 clean cancel 且不丢中间结果——用 `task.cancel()` + `CancelledError` 捕获 + `finally` 块标记状态解决。

- **R**：最终 Leader 可以同时派发多个子 Agent 并行工作，用户等待期间能继续跟 Leader 交互。

---

### 题 2：「如果重新做这个项目，你会怎么改进？」

- **S**：第一次做企业级 Agent 平台，很多决策在边学边做中定下来。

- **A**：
  - **评测体系前置**：没有离线评测集，说不出 RAG 检索的 Recall 到底是多少。应该早建 golden dataset，做自动化回归评测。
  - **可观测性**：目前只有日志没有结构化 trace。应该接入 OpenTelemetry，让每次 LLM 调用、工具调用都能在一条 trace 上回溯，排障从「翻日志」变成「看 trace」。
  - **API 契约管理**：5 个微服务之间的接口是口头约定的。应该用 OpenAPI / Protobuf 做 schema 管理，减少联调对接成本。

- **R**：不影响当前架构方向，但能显著降低中后期迭代风险。系统演进必然迭代，关键是技术债可控可还。

## 四、面试禁忌提醒

1. **不要只讲技术名词**：每次讲完一个技术点，补一句「这解决了什么问题」或「用户/业务感受到了什么变化」。
2. **区分贡献层级**：你是独立开发，面试时用「我负责」「我设计」「我实现」而不是「我们做」。
3. **诚实面对量化数据**：项目没有线上用户，直接说「这是独立开发项目，本地测试了 X 个场景」，不编造业务数据。
4. **不要背书式回答**：记 STAR 结构 + 3~5 个关键词，用口语串联。被追问时先确认层级（架构/实现/指标/取舍）。
5. **准备反向提问**：建议问「团队目前 Agent 开发最头疼的是评测还是稳定性？」或「Agent 的错误恢复机制是怎么做的？」
